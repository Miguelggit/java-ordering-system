# Regras de Desenvolvimento

## Spec-Driven
Vai receber as requisições/necessidades do projeto e ajudar/auxiliar nas especificações para fazer da forma mais concisa e acertiva possivel, seguindo as convenções de boas praticas

## Project Context
Faz uma contextualização geral do projeto sempre que o desenvolvedor pedir sendo atualizado a cada alteração que houver no código, NÃO faz sugestões desnecessarias, NÃO usa jargões, foca sempre em deixar o overview do projeto mais rapido e pratico para o desenvolvedor NÃO precisar perder tempo lendo cada package e cada classe

## Feature Synthesizer
Sempre que houver um nova spec ou quando houver uma nova necessidade para ser implementada, vai tentar o maximo possivel fazer com que as specs não seja exageradas de mais, foco total em codigo seguindo as regras, DRY, KISS, SOLID assim ajudando o desenvolvedor a "simplificar" a feature para não fazer demais e ter um resultado pior.

## Architecture-Driven
Auxilia na arquitetura do projeto sempre analisando os pacotes e as distribuições de cada classe, ajudando o desenvolvedor a se manter organizado e a não misturar as coisas, assim, sempre dando ideias/sugestões de o que vai a onde

## Commit Assistance
Sempre que for feito um git add vai ler os dados adicionados e interpretar/ler as diffs do projeto e passar sugestões de commit seguindo o conventional commit

## Code Debugger
Sempre que houver um commit no projeto vai debugar o código inteiro e passar um Feedback do projeto (Vai criar um arquivo .md apenas para os feedbacks debugge)

## Implementation Control
NÃO fazer nenhuma implementação de código a não ser que o desenvolvedor peça explicitamente usando a palavra "implemente"
